# Michael Product Lab

A bilingual product portfolio that turns Michael's public and private GitHub repositories into product case cards.

The app is built with Next.js, TypeScript, Tailwind CSS, GSAP, and a static data pipeline based on the authenticated GitHub CLI.

## What It Shows

- All curated GitHub repositories as product cards
- zh-TW / English language toggle
- Product-line filtering
- Featured product section
- Created and last-updated metadata
- Real preview screenshots for deployed projects
- Fallback preview assets for projects without public demos

Private repositories are displayed as product cases, but source links are not exposed in the UI.

## Scripts

```bash
npm run dev
npm run lint
npm run build
npm run sync:repos
npm run sync:previews
npm run sync:all
```

`sync:repos` uses the local `gh` session, so it requires an authenticated GitHub CLI with access to Michael's repositories.

`sync:previews` uses Playwright Chromium to capture homepage screenshots when a repo has a reachable `homepageUrl`. Repos without a reachable homepage receive generated fallback previews.

## Data Files

- `src/data/repos.generated.json` is generated from GitHub metadata.
- `src/data/preview-manifest.generated.json` is generated by the preview capture script.
- `src/data/project-overrides.json` is the hand-curated product layer for category, bilingual copy, features, featured status, and sorting.

## Architecture Boundaries

The app keeps responsibilities separate so product content, generated data, and UI behavior do not interfere with each other.

- `scripts/` owns data generation and preview capture.
- `src/data/` stores generated metadata and curated product overrides.
- `src/lib/projects.ts` is the project catalog read model. UI components read normalized `Project` objects instead of touching raw JSON.
- `src/components/PortfolioExperience.tsx` owns page-level state only: locale, active category, and selected project.
- `src/components/portfolio/` owns presentational sections and small UI pieces.

If CRUD editing is added later, create/update/delete operations should live behind a data layer or API route. Cards, modals, and sections should stay display-only.

## Development

```bash
npm install
npx playwright install chromium
npm run sync:all
npm run dev
```
